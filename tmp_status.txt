What is already working:
- Fresh Claude sessions can load the project-local `c2c` MCP server from `.mcp.json`.
- Fresh PTY-launched sessions can be discovered reliably once auto-primed.
- The MCP server declares `experimental["claude/channel"]`.
- `mcp__c2c__list` works in fresh live sessions.
- `mcp__c2c__whoami` works in fresh live sessions.
- `mcp__c2c__send` works in fresh live sessions and returns `queued`.
- Python CLI registration works for live sessions.
- Broker enqueue works.
- Broker drain works.
- `poll_inbox` is implemented in the OCaml MCP path and committed.
- A polling client can disable automatic post-RPC channel drain with `C2C_MCP_AUTO_DRAIN_CHANNEL=0` and then receive messages explicitly through `poll_inbox` tool output.
- The Python-side broker registry sync preserves broker-only registrations such as `codex-local` instead of overwriting them.
- Codex restart/resume launcher support exists via `run-codex-inst` and `run-codex-inst-outer`, with per-instance `C2C_MCP_SESSION_ID` injection.
- A real Codex participant is currently running as broker session `codex-local` with alias `codex`.
- Live broker/tool-path proof has now been achieved for the pull route: a live Claude peer sent to alias `codex`, the running Codex participant drained via MCP polling, and the sender side observed the inbox drain/ack cycle.
- The rebuilt OCaml broker is now live in at least the active storm-echo session; `sweep` was run and cleaned 121 orphan inbox files with zero dropped registrations.
- MCP initialize response now includes explicit `instructions`.
- MCP initialize now reports the server-supported protocol version instead of blindly echoing arbitrary client versions.
- Broker registry mirroring is now atomically written under the registry lock.

What is not yet working:
- Receiver-side transcript-visible delivery of inbound `notifications/claude/channel` messages is still not working in live Claude sessions.
- The currently active `r2` pair cannot prove or disprove that path because it was launched without the required development-channel flags.
- Because of that, real visible two-way autonomous chat between live Claude sessions is not yet proven.

Recent hardening completed:
- The Python CLI send path now supports broker-only peers like `codex` by falling back to the broker registry when an alias is not present in the YAML/live-Claude registry.
- Broker-only fallback sends now use the caller's registered alias when available, instead of always stamping `from_alias` as `c2c-send`.
- Broker-only inbox appends in `c2c_send.py` are now serialized with a per-inbox thread lock plus POSIX `fcntl.lockf` sidecar lock and atomic replace, so concurrent Python sends do not drop messages and Python interlocks with OCaml `Unix.lockf`.
- Python broker sync now preserves existing liveness metadata for YAML-backed peers, and Python's broker-only CLI fallback now rejects dead peers instead of appending into orphan inboxes.
- `c2c-poll-inbox` now gives Codex a local recovery path when host MCP tools are absent: it tries direct JSON-RPC polling first, then falls back to draining `<session>.inbox.json` under the OCaml-compatible `<session>.inbox.lock` sidecar if MCP startup fails.
- The recovery poller is now part of the normal operator surface: `c2c poll-inbox ...` dispatches to it, and `c2c install` installs `c2c-poll-inbox`.
- `c2c inject` / `c2c-inject` now provides a one-shot PTY injection surface shared by Claude (`--claude-session`), Codex (`--pid`), and OpenCode/generic terminals (`--terminal-pid --pts`), with `--dry-run` + `--json` for safe probing.
- `c2c deliver-inbox` / `c2c-deliver-inbox` now bridges broker inboxes to live terminals: it peeks safely in `--dry-run`, otherwise drains a session inbox and injects each queued C2C message into Claude/Codex/OpenCode via the shared PTY injector.
- `c2c deliver-inbox --loop` now provides an always-on broker-to-PTY delivery loop for Claude, Codex, OpenCode, and generic terminals. `--max-iterations` bounds tests/probes, `--interval` controls polling cadence, and `--pidfile` gives operators a concrete process marker.
- `c2c deliver-inbox --daemon --loop --pidfile ...` now starts that live delivery loop as a managed detached process, reuses an already-running pidfile instead of duplicating loops, and reports the daemon/log paths as JSON or text.
- `restart-codex-self --reason ...` now writes `run-codex-inst.d/<name>.restart.json` before signaling the managed Codex pid, so the resumed turn has local context even if C2C MCP startup failed.
- The Python follow-up slice is reverified green, including a launcher dry-run assertion that is now interpreter-path agnostic.
- The Codex PTY poker loop is running with an orientation heartbeat that polls inbox, reads status/locks when needed, treats empty inbox as not-a-stop-signal, and continues unblocked work.
- `c2c_poker.py --pid ...` and `--claude-session ...` now watch the original client/session pid after resolving terminal coordinates and exit cleanly if that pid goes away. Poker heartbeat payloads include a fresh `Sent at: ...` timestamp/date on each injection.
- `c2c_mcp.py` now builds the OCaml server explicitly and launches the built executable directly instead of running through `bash -lc ... dune exec ...`, reducing the documented wrapper-chain leak risk in the MCP launcher path.
