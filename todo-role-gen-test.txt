## Testing process for agent-file epic

The epic ships in stages. At each stage, the expected behavior is
spelled out below so we can verify independently (Max + swarm dogfood).

### Stage 1 — Parser & renderer fidelity (blocks all downstream)

**What**: `c2c_role.ml` parses canonical `.c2c/roles/<name>.md` and
renderers emit correct per-client agent files with ALL fields preserved.

**Fixtures**:
- `.c2c/roles/coordinator1.md` (already seeded; has `c2c.auto_join_rooms`,
  `claude.tools`, `opencode.theme`)
- `.c2c/roles/ceo.md`, `.c2c/roles/galaxy-coder.md`,
  `.c2c/roles/jungle-coder.md` (seeded)

**Expected**:
- `c2c roles --dry-run <name>` for opencode target includes ALL
  `c2c:` fields including arrays (`auto_join_rooms`)
- `c2c roles --client claude --dry-run <name>` includes ALL `claude:`
  fields including arrays (`tools`)
- No silent field drops. No YAML syntax errors in output.
- Body passes through verbatim (no markdown munging).

**Current status** (2026-04-22, b748257 + 426c852):
- ❌ Arrays dropped (todo.txt line 45) — ceo traced to `c2c_role.ml:68`
  and `:75`
- Text fields pass through correctly
- Body passthrough looks correct

**Exit criteria**: all seeded fixtures render losslessly for both
opencode and claude targets.

### Stage 2 — `c2c agent new` template quality

**What**: creating a new role via `c2c agent new <name>` produces a
useful starter template (not empty YAML keys).

**Expected**:
- Output file has **either** no empty keys OR commented example stubs
- Template includes a placeholder body with a prompt-ish sentence
  (currently "You are a `<name>` agent." — OK but could be richer)
- File is valid YAML that `c2c_role.ml` can re-parse without warnings

**Current status**: ❌ sparse template with empty `model:`, `c2c:`,
`opencode:` keys (todo.txt line 46).

**Exit criteria**: `c2c agent new foo && c2c roles --dry-run foo` round-trips
without warnings and produces sensible commented guidance for editors.

### Stage 3 — Banner rendering

**What**: substantial commands print a themed ASCII-art banner header
on invocation (per Max's direction).

**Expected**:
- `c2c agent new <name>` shows a banner with the command name,
  timestamp, and product tagline before any other output.
- `c2c start <client> --agent <name>` shows a banner themed by the
  role's `opencode.theme` or falls back to `default`.
- `c2c roles --dry-run` shows a banner before the per-role output blocks.
- Banner uses accent+muted colors from the theme. 11 themes wired.

**Current status**: ❌ Banner.ml exists but is never invoked — output
is plain `=== name (client) ===` text (todo.txt line 47).

**Exit criteria**: each substantial command prints the themed banner;
screenshot or recorded terminal matches Max's taste bar.

### Stage 4 — End-to-end launch (OpenCode)

**Expected**:

- OpenCode starts and the agent is selected (TUI shows it)
- Theme overlay applies (TUI visual check)
- Agent registers to broker with the right alias
- Auto-joins the rooms listed in `c2c.auto_join_rooms`

**Current status**: 🟡 mechanism exists per b748257 commit message;

**Exit criteria**: Max confirms live session launches cleanly with

## Schema v1.1 extensions (post-MVP, Max 2026-04-22)

Add to canonical `.c2c/roles/<name>.md` frontmatter:

- [ ] **`compatible_clients: [opencode, claude]`** — whitelist of
      clients this agent supports. Launcher rejects mismatches:
      `c2c start codex --agent visual-analyst` aborts with clear error
      if `codex` isn't listed. Default: all clients if omitted.

- [ ] **Per-client `model:` override** — `opencode.model:
      anthropic/claude-sonnet-4-20250514` and `claude.model: opus`
      already possible per OpenCode's schema; formalize as first-class
      v1.1 field + document in docs/agent-files.md.

- [ ] **`required_capabilities: [vision, tools, reasoning]`** —
      declarative capability requirements. Launcher checks client's
      configured model against this list before launch; aborts early
      if mismatch. Useful for: visual-analyst (vision), code-agents
      (tools), planner roles (reasoning). Schema: free-form strings;
      launcher maps to per-model capability tables.

Use case driver: Max noted that we "often want a specific model for a
specific thing (like we need multimodal ones for visual analysis)".
Without this, users launch vision-requiring agents with non-vision
clients and hit runtime errors — bad UX. Compile-time rejection is
better.

- [ ] **`include: [<snippet>, ...]`** — snippet imports. Max 2026-04-22:
      "our agent files should support importing/including snippets. like
      we can have a default snippet about how to use c2c, and what are
      good monitors or scheduled tasks to set up." Proposal:
      ```yaml
      ---
      description: My agent
      include: [c2c-basics, monitors-setup]
      ---
      body goes here
      ```
      Loader resolves each name to `.c2c/snippets/<name>.md`, concatenates
      (prepend-to-body or inject-at-marker, TBD) into the compiled output's
      body. Allows DRY'ing common stanzas across many roles (how to use
      c2c, how to arm monitors, push discipline, finding-log conventions).
      Schema location: top-level frontmatter field (not per-client).
      Expansion happens before per-client rendering so all clients get
      the same body. Initial snippet set: `c2c-basics.md` (DMs + rooms +
      poll_inbox), `monitors-setup.md` (c2c monitor --all + heartbeat +
      ScheduleWakeup patterns), `push-policy.md` (coordinator gate). We
      already have 4 canonical roles that could refactor to use these
      once the loader lands.
