## 8. Lessons (cross-cutting, this burn)

1. **Review-and-fix is a burn driver, not just a quality gate.** When the coordinator dispatched 8+ docs slices in parallel and review-and-fix turned each FAIL into a same-window respin (#340a, #361), the loop *increased* throughput rather than slowing it — because the alternative was a deferred fix tomorrow. The skill earns its keep when invoked aggressively, not sparingly.

2. **Parallel-subagent dispatch pattern works for docs, but bites OCaml.** Coordinator running 4–5 subagents in parallel on independent doc files (#348, #356–#359, #369) was clean. The same pattern on OCaml builds softlocked dune. **Heuristic going forward**: parallel-OK for read/write-disjoint docs slices; serialize anything that touches `_build/` or runs `just`.

3. **Peer-PASS is a process helper, not a security gate.** Max's calibration today made this explicit: substantive review is the whole point; sealing is bookkeeping. The `--allow-self` allowance for mechanical/low-stakes slices (#371) is a deliberate easing — it acknowledges that not every doc-typo needs a fresh-slate reviewer. The discipline lives in *what review means*, not in *who signs*.

4. **Dogfood loops close via meta-self-application.** #323 (auto-DM-on-cherry-pick) was tested by *coordinator cherry-picking the #323 fixup itself and receiving the auto-DM*. That's the cleanest possible end-to-end validation: the feature observed itself working in production while it was being hardened. Going forward, every new coord-workflow feature should be designed so its first user is the coordinator delivering it.

5. **Document the regression class while it's hot.** Four subagent failure modes hit in two hours; they all got runbook lines (#373) before the window closed. If we'd waited for a retrospective, three of the four would have been re-hit by the next agent.
