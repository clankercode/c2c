4. **Kill ideas that don't fit**: if the swarm decides an idea is out of scope, mark `ingested` with a note explaining why (so it doesn't re-surface).

Idea: PoW-gated email proxy for agent commit attribution at c2c.im

Submitted by: Max Kaye (via coordinator1) · 2026-04-24T17:02Z
Status: brainstorming and planning
Related: git attribution convention (CLAUDE.md), todo.txt #119, swarm-lounge discussion 2026-04-24T11:53 AEST

Context:
Agent commits should use `--author="<alias> <alias@c2c.im>"` so:
1. attribution is clear in git log (who on the swarm did what)
2. the email is real advertising for c2c.im

To make the email genuinely contactable without enabling spam, run a PoW-gated
email proxy: sender must submit a proof-of-work to deliver a message, so bulk
spamming is economically unattractive but a single interested human can always
reach through.

Immediate slice (SHIPPED 2026-04-2x — git.attribution config + `--author` injection):
- ✅ `git.attribution` config schema landed (`ocaml/c2c_start.ml:498` `repo_config_git_attribution`)
- ✅ `c2c git` shim wires `--author="<alias> <alias@c2c.im>"` (`ocaml/cli/c2c.ml:2081`)
- ✅ Placeholder alias@c2c.im in use; bounces are fine pending PoW proxy
- ✅ CLAUDE.md convention documented

Future slice (PoW proxy) — design decisions locked 2026-04-28 by Max:
- Stand up email proxy at c2c.im that accepts mail only after valid PoW header
- Wire MX record; any alias@c2c.im routes through proxy
- **PoW scheme**: hashcash with **dynamic difficulty**. Each agent sets its own
  difficulty target (advertised via DKIM-style header or profile-page lookup).
  Senders compute hashcash to match the recipient agent's target.
- **Bounceback on insufficient PoW**: emails that fail to meet the target
  difficulty get a bounce reply with the required difficulty + sample
  hashcash invocation, so legitimate senders can retry.
- **Recipient routing**: per-alias inbox routing (each alias gets its own
  inbox at the proxy; cross-references with c2c profile page for the
  optional link-back).
- **Profile-page link**: yes — emails that pass PoW carry a header/footer
  pointing to the sender agent's public c2c profile (depends on c2c GUI
  profile page progress).

Idea: `c2c list -g` / `--global` — show every running c2c managed session system-wide (across repos / brokers), not just the current repo's broker.

Submitted by: Max Kaye (via coordinator1) · 2026-05-01T06:11Z
Status: **ingested** — shipped by galaxy-coder (63dc95c6 + 342c8679). `c2c list -g` scans all broker roots, groups by broker. Closed.
Related: #517 (c2c list shows tmux session:window.pane), broker-root resolution (CLAUDE.md "Key Architecture Notes")

Context: users may forget about c2c instances they started in another repo / another broker root. `c2c list` today only enumerates registrations under the resolved broker root for the current cwd. A `-g`/`--global` flag would scan all known broker roots (e.g. `$HOME/.c2c/repos/*/broker/` + any `$XDG_STATE_HOME/c2c/repos/*/broker/`) and surface everything currently registered, regardless of repo. Useful for finding orphaned outer-loops, rogue notifiers, or sessions started in a different worktree.

Open questions:
- Discovery scope: only canonical `~/.c2c/repos/*/broker/` + XDG, or also include legacy `<git-common-dir>/c2c/mcp/` paths flagged "legacy"?
- Output: per-broker subsections, or a flat list with a `repo:` column?
- Liveness check parity with single-broker `c2c list` (which already classifies alive/dead via /proc).
- Should it also pick up running outer-loop wrappers that haven't registered yet (`pgrep -af 'c2c start'`)?

Idea: OpenCode heartbeat should suppress when agent is active (send only when idle)

Submitted by: Max Kaye (via coordinator1) · 2026-05-02T06:38Z
Status: new
Related: agent-wake-setup runbook, heartbeat Monitor, OpenCode statefile

Context:
The heartbeat tick (4.1m cadence) fires unconditionally for OpenCode peers.
When the agent is actively generating (statefile shows active/in-progress),
the heartbeat notification is wasted — it interrupts the model mid-generation
or queues a redundant wake. The heartbeat should only fire when the agent is
idle, so it acts as a true "poke the idle agent" mechanism.

Open questions:
- Where does OpenCode expose idle/active state? Statefile path + format?
- Should the heartbeat binary itself check the statefile, or should the
  outer-loop wrapper gate heartbeat delivery?
- Does this apply to other clients (Claude Code, Codex) or only OpenCode?
- Edge case: agent is "active" but stuck on a permission prompt — heartbeat
  is still useful there. Need to distinguish generating vs waiting-for-input.

**Update (Max 2026-05-02T07:00Z):** Superseded by broader idea below — c2c-native
self-scheduling. This idle-gating becomes `only_when_idle` flag on the schedule.

Idea: c2c-native agent self-scheduling (replaces external heartbeat/Monitor)

Submitted by: Max Kaye (via coordinator1) · 2026-05-02T07:00Z
Status: new
Related: heartbeat binary, Monitor tool, agent-wake-setup runbook, idle-gating idea above

Context:
Since c2c can send messages to agents, agents can schedule their own wake-ups
via c2c self-DMs on a timer. State persisted in `.c2c/schedules/<alias>/`.
Eliminates the external `heartbeat` binary + Monitor tool dependency entirely.
Agents control their own cadence; schedules survive compaction (on-disk).

Proposed shape:
```
c2c schedule set --interval 4.1m --message "wake — poll inbox" --only-when-idle
c2c schedule set --interval 1h --align :07 --message "sitrep tick"
```

Design decisions needed:
- Timer host: MCP server background thread (long-lived, sees tool calls) vs
  `c2c start` outer wrapper vs standalone daemon
- Idle detection: "no tool call in last N seconds" (MCP server has this signal)
  vs client-specific statefile vs both
- `only_when_idle` semantics: skip delivery entirely, or queue and deliver on
  next idle transition?
- Cross-client parity: OpenCode statefile vs Claude Code tool-call cadence vs
  Codex activity signal
- CLI surface: `c2c schedule {set,list,rm}` subcommand group

Additional trigger: file-change detection
- Schedule config lives in `.c2c/schedules/<alias>/` — the MCP server (or
  daemon) should watch this path with inotify. When an agent updates its
  own schedule (e.g. `c2c schedule set ...`), the timer reconfigures
  automatically without restart.
- Extends naturally: watch `.c2c/config.toml` changes too, so agents can
  hot-reload their own configuration.
- Could also watch inbox files for size changes (replacing the current
  inbox watcher polling loop with inotify).

Benefits:
- No Monitor re-arming on restart — schedule is on disk
- Idle-gating built in — no wasted heartbeats during active generation
- Agent self-governance — each agent controls its own wake cadence
- Eliminates heartbeat binary dependency
- File-change autodetect — schedule updates take effect immediately
