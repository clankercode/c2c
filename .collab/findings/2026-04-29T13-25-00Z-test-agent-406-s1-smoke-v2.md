## Finding 4: `setUpClass` `--build` context check fails on unreadable volume files

**Symptom**: All 6 `DockerExecIntegration` tests still skip after v2 fixes.

**Root cause**: `setUpClass` calls `docker compose up -d --build`. The `--build` flag makes Docker scan the **build context** (the compose file's `context: .`) for all files. The `volumes/` directory is bind-mounted at `./volumes/relay-a` and `relay-b`. When the relay container first runs, it creates files (sqlite DB, identity JSON) owned by uid=999 (the container's `c2c` user). Those files are unreadable by the host's docker build process.

```
checking context: no permission to read from '.../volumes/relay-a/relay-server-identity.json'
```

`compose up --build` exits rc=1 → `setUpClass` calls `compose down` and raises `SkipTest`.

**Workaround applied**: Manual `up -d` (without `--build`) after `chmod 777 volumes/` works fine. The images were already built.

**Fix options**:
1. **Preferred**: Change `setUpClass` to `up -d` without `--build` (images already exist from CI/pre-build step). Add a separate `test_force_rebuild` if rebuild is needed.
2. Add `volumes/` to `.dockerignore` in the worktree.
3. Use named Docker volumes instead of host bind mounts (more isolated but changes the test topology).

## Finding 5: `test_c2c_binary_exists_in_thin_agent` assertion checks for "c2c" in `--version` output

**Symptom**: Binary exists and returns version correctly, but test assertion fails.

The test asserts:
```python
self.assertIn("c2c", r.stdout.lower())
```

But `c2c --version` returns: `0.8.0 2026-04-29T13:22:31Z`

There is no "c2c" string in the version output — this assertion would always fail on the OCaml binary.

**Fix**: Change assertion to check return code == 0 and that the output contains a version number pattern, or check for "c2c" in `c2c --help` output instead.
