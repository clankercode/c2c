## Finding 1: `_wait_for_health` uses `sh` + `/dev/tcp` (bash-only)

**Symptom**: All 6 `DockerExecIntegration` tests skip — `setUpClass` raises `unittest.SkipTest` at the health-check step.

**Root cause**: `_wait_for_health` at line 59 uses:
```python
sh -c, f"exec 3<>/dev/tcp/127.0.0.1/{port} ..."
```
`/dev/tcp` is a bash builtin; `sh` (typically dash on Linux) does not support it.

**Evidence**:
```
docker exec c2c-relay-a sh -c 'exec 3<>/dev/tcp/127.0.0.1/9000 ...'
# → sh: 1: cannot create /dev/tcp/127.0.0.1/9000: Directory nonexistent
```
Health check returns rc=2 → `setUpClass` calls `compose down` and skips all tests.

**Fix**: Change `sh -c` to `bash -c` in `_wait_for_health` (bash is available in the relay container at `/usr/bin/bash`).

## Finding 2: Cross-relay send uses bare alias but OCaml relay requires `alias@host`

**Symptom**: `test_send_alice_to_bob_via_relay_a` fails with:
```
fatal: not a git repository ...
error: unknown alias: bob   (rc=1)
```
Expected: rc=0 (queued to relay, dead-lettered).

**Root cause**: The OCaml relay validates aliases against its local registry. `bob` is registered on `relay-b`, not `relay-a`. When alice (relay-a) sends to bare `bob`, relay-a rejects with "unknown alias" before any queue/dead-letter action.

The correct cross-relay format is `bob@host-b`:
```
alice@relay-a → c2c send bob@host-b "msg"  → rc=0, ok -> bob@host-b
```
**Evidence**:
```
docker exec c2c-peer-a1 c2c send bob 'Hello'    → rc=1, unknown alias: bob
docker exec c2c-peer-a1 c2c send bob@host-b 'Hello' → rc=0, ok -> bob@host-b
```

The test comment says the message goes to dead-letter on relay-a, but the OCaml relay errors out at send time instead. This is a test assumption mismatch — the test was likely written against a Python relay that silently dead-lettered unknown aliases, or the OCaml relay behavior is itself a deviation.

**Fix**: Either (a) use `bob@host-b` format in the test, or (b) register bob on relay-a too for local delivery, or (c) update the test comment to reflect the actual error behavior and change the assertion to expect rc=1.

## Finding 3: `--relay-url` flag doesn't exist in OCaml CLI

**Symptom**: Tests pass `--relay-url http://relay-a:9000` but the OCaml `c2c send --help` shows no `--relay-url` flag.

**Root cause**: `--relay-url` is a Python CLI flag. The OCaml CLI uses the `C2C_RELAY_URL` environment variable (which is already set correctly in the compose file for all peer containers).

**Fix**: Remove all `--relay-url` arguments from test invocations. The compose file sets `C2C_RELAY_URL` in each peer container's environment, so the binary automatically uses the correct relay.
