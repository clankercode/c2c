## HIGH — communication reliability

### P1. Storm-echo routing black-hole (orphan-alias regs)

**Symptom:** Every message I sent to alias `storm-echo` via `mcp__c2c__send`
was enqueued to a dead session's inbox file. Storm-echo never saw any of my
acks or status notes.

**How discovered:** Broadened Monitor noticed my sends hitting
`92568b24-8b11-….inbox.json` instead of a live file. Dumped registry.json
and saw TWO legacy pid=None regs sharing alias `storm-echo` (session_ids
92568b24 and 9d0809b5). Dumped both inboxes and found **5+ undrained
messages** — not just mine, also 2 acks from codex.

**Root cause chain:**
1. Storm-echo was re-launched multiple times, each run creating a new row in
   `registry.json` via `Broker.register`.
2. `register` pre-fix dedupes by `session_id` only, not by `alias`, so every
   re-launch leaves the previous row in place.
3. `registration_is_alive` treats pid=None (legacy) as ALIVE for backwards
   compat, so the ghost row survives `sweep`.
4. `enqueue_message` resolves alias → first **live** match. Ghost + real
   session both "alive" → first in list wins → ghost wins every time.
5. Messages accumulate in the ghost inbox forever.

**Severity:** HIGH — this is exactly the "unify all agents" primary goal
and it silently failed without any error to the sender.

**Fix (uncommitted, working tree):** `Broker.register` now dedupes by BOTH
`session_id` and `alias`. Re-registering alias `storm-echo` atomically
evicts all prior rows for that alias. 30/30 tests green including new
`register evicts prior reg with same alias`.

**Retrofit caveat:** Pre-existing ghost rows can't be fixed until the new
binary is restarted AND the stranded participant re-registers. Someone
(likely Max) manually cleaned the 3 ghost rows in this case — but the 5+
stranded messages are still sitting in the old inbox files until sweep runs
after a restart.

## MEDIUM — cross-language race

### P5. Python broker-only fallback had the same inbox RMW race

**Symptom:** Codex's `c2c_send.py` broker-only path read+append+write the
inbox file without any lock, mirroring the OCaml pre-lockf shape.

**How discovered:** Wrote a Python fork test to reproduce the race
empirically. 12 children each append 20 messages to a shared JSON:
- **Without lock:** lost 237/240 on trial 0, 224/240 on trial 1, then
  `json.decoder.JSONDecodeError: Extra data` on trial 2 (partial write
  clobbered another child's read).
- **With `fcntl.lockf(LOCK_EX)` wrapping RMW:** 240/240 × 5 clean.

**Fix status:** OCaml side fixed (`Broker.with_inbox_lock` in working
tree, 29/29 tests + stable under repeats). Python side landed at 13:32
but see **P11** — codex used `fcntl.flock` (BSD) while OCaml uses
`Unix.lockf` (POSIX fcntl). These DO NOT interlock on Linux. My earlier
"POSIX fcntl, interlocks cleanly" claim was wrong. Single-language
protection holds; cross-language protection does not.

### P11. Cross-language lock mismatch: flock vs lockf are independent on Linux

**Symptom:** OCaml broker uses `Unix.lockf` (POSIX fcntl advisory),
Python `c2c_send.py` uses `fcntl.flock` (BSD-style). On Linux local
filesystems these are held in **independent kernel lock tables** — a
process holding one does not block a process taking the other on the
same file.

**How discovered:** Reviewing c2c_send.py:152 after codex's 13:32
release, spotted `fcntl.flock(handle.fileno(), fcntl.LOCK_EX)`. Wrote
a tiny probe in /tmp:

```
parent: fcntl.flock(fd, LOCK_EX)   # hold
child:  fcntl.lockf(fd2, LOCK_EX|LOCK_NB)
result: "CHILD: lockf acquired DESPITE parent flock"
```

Confirmed: the locks are independent. A concurrent OCaml broker write
and Python broker-only send against the same inbox file would race
exactly as if neither had a lock, even though both SIDES are internally
race-free.

**Severity:** HIGH — this is a silent correctness hole directly in the
storm-beacon ↔ codex interop path, which is the whole point of the
broker-fallback slice. Single-language stress tests pass, but any
workload that mixes OCaml MCP writes and Python CLI sends on the same
inbox can lose messages.

**Proposed fix (minimal):** Switch Python to `fcntl.lockf(handle,
LOCK_EX)` — a one-line change in `broker_inbox_write_lock` at
`c2c_send.py:152`. POSIX fcntl advisory locks DO interlock across
processes regardless of language, so my OCaml `Unix.lockf` path will
then see Python contention and vice-versa.

**Alternate fix:** Add an OCaml C stub for `flock(2)` and use it in
`Broker.with_inbox_lock`. More invasive; no benefit over fixing Python.

**Status:** Alerted codex + storm-echo via c2c message at 13:33.
Codex claimed `c2c_send.py` at 13:43 with purpose "switch broker inbox
lock from flock to POSIX lockf" — fix in flight. Tracking.
