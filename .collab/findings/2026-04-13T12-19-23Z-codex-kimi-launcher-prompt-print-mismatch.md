# Kimi managed launcher prompt/print mismatch

- **Symptom:** `run-kimi-inst` launched configured Kimi instances in one-shot print mode whenever the config contained `prompt`. The outer launcher also started Kimi in a new session, leaving the process with terminal fds but no controlling TTY/job-control link.
- **How discovered:** Max reported a detached Kimi process and asked whether the managed launcher lined up with the Kimi CLI for interactive mode. `kimi --help` on Kimi CLI 1.31.0 shows `--prompt` and `--print` are independent flags; `RUN_KIMI_INST_DRY_RUN=1 ./run-kimi-inst kimi-nova` showed the wrapper emitted `kimi --yolo --print --prompt ...`. Process inspection showed live Kimi had stdin/stdout on `/dev/pts/0` but process/session IDs detached from that terminal's foreground process group.
- **Root cause:** The wrapper encoded `prompt` as "non-interactive print prompt" instead of "initial prompt". The outer restart loop also used `subprocess.Popen(..., start_new_session=True)`, which is useful for daemon-like children but wrong for an interactive Kimi TUI.
- **Fix status:** Fixed in this session by making `prompt` pass `--prompt` while staying interactive, adding explicit config `"print": true` for one-shot runs, and changing `run-kimi-inst-outer` to keep the Kimi child in the launching terminal session.
- **Severity:** Medium. It made managed Kimi sessions detach from the expected interactive frontend and made Ctrl-C/job-control behavior confusing.
