# Dune lock corruption and managed-env leak in copied-checkout test

- **Symptom:** `just test` initially failed during `dune build` with `Command exited with code 2` while loading the OCaml compiler context. A verbose rerun exposed `Error: Unexpected contents of build directory global lock file (_build/.lock). Expected an integer PID. Found:`.
- **How discovered:** `opam exec -- dune build ... --verbose` gave only the compiler-context failure. `--display=verbose` plus inspecting `_build/.lock` showed the lock file was zero bytes. Removing `_build/.lock` allowed a normal build again.
- **Second symptom:** After the build recovered, Python failed in `C2CCLITests.test_list_returns_recently_registered_sessions_in_same_environment`; the copied checkout's `c2c-register` tried to create `checkout-register/c2c/mcp` and crashed with `NotADirectoryError` because `checkout-register/c2c` is the wrapper file.
- **Root cause:** That test built a custom env without an explicit `C2C_MCP_BROKER_ROOT`. `c2c_register.sync_broker_registry_from_env()` always syncs the broker after updating the YAML registry. In the copied checkout, git-common-dir fallback resolved to `checkout-register/c2c/mcp`; `checkout-register/c2c` is the copied wrapper file, not a directory. The test also did not clear managed auto-register/join vars, so it was more exposed to the managed-agent environment than adjacent tests using `self.env`.
- **Fix status:** Deleted the stale `_build/.lock` build artifact and patched the copied-checkout test env to set a shared temp `C2C_MCP_BROKER_ROOT` and blank the managed auto-register/join vars.
- **Severity:** Medium. The build lock issue blocks verification with an opaque Dune error; the env leak makes tests fail only inside managed agents, which is exactly where the swarm runs them.
