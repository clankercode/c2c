# Kimi managed launcher prompt/print mismatch

- **Symptom:** `run-kimi-inst` launched configured Kimi instances in one-shot print mode whenever the config contained `prompt`. The outer launcher also started Kimi in a new session, leaving the process with terminal fds but no controlling TTY/job-control link. After the first fix, top-level `kimi --prompt ...` still produced formatted non-TUI output instead of the interactive frontend. A later attempted fix appended `term`, which launched Toad rather than Kimi Code CLI. After removing `term`, Kimi loaded correctly but the configured prompt was not visible.
- **How discovered:** Max reported a detached Kimi process and asked whether the managed launcher lined up with the Kimi CLI for interactive mode. `kimi --help` on Kimi CLI 1.31.0 shows `--prompt` and `--print` are independent flags; `RUN_KIMI_INST_DRY_RUN=1 ./run-kimi-inst kimi-nova` showed the wrapper emitted `kimi --yolo --print --prompt ...`. Process inspection showed live Kimi had stdin/stdout on `/dev/pts/0` but process/session IDs detached from that terminal's foreground process group. `kimi term --help` says it runs Toad backed by Kimi Code CLI ACP, and Max's managed run printed Toad's startup/shutdown UI. PTY smoke confirmed top-level `kimi --yolo` starts Kimi Code CLI interactively. Inspecting installed Kimi source showed `KimiCLI.run_shell(..., prefill_text=...)` and `CustomPromptSession.set_prefill_text(...)`, but no public CLI flag for shell prefill.
- **Root cause:** The wrapper encoded `prompt` as "non-interactive print prompt" instead of separating interactive and one-shot modes. The outer restart loop also used `subprocess.Popen(..., start_new_session=True)`, which is useful for daemon-like children but wrong for an interactive Kimi process. I then misread `kimi term` as Kimi's interactive entrypoint; it is the Toad TUI wrapper. Kimi's public `--prompt` path calls `Shell.run(command)` and exits after that command, while editable prefill is an internal API.
- **Fix status:** Fixed by adding explicit config `"print": true` for one-shot runs, changing `run-kimi-inst-outer` to keep the Kimi child in the launching terminal session, and correcting `run-kimi-inst` so non-print interactive launches do not use `term`. Configured interactive `prompt` now launches through `c2c_kimi_prefill.py`, a small shim run with Kimi's own Python interpreter that maps public `--prompt` to Kimi's internal shell prefill path.
- **Severity:** Medium. It made managed Kimi sessions detach from the expected interactive frontend and made Ctrl-C/job-control behavior confusing.
