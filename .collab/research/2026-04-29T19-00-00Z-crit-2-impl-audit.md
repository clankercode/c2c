The one fact that surprises a fresh read of the design: it describes
the per-alias storage shape (§2.1) as a nested record with
`first_seen_ts` / `last_seen_ts`. The actual persistence shape in
`relay_pins.json` is **flat top-level sections** (`x25519`, `ed25519`,
`min_observed_envelope_versions`) keyed alias → value. No
`first_seen_ts` / `last_seen_ts` are stored. This is a documentation
delta, not a security defect — the design's described fields are
*nice-to-have observability surface*, not load-bearing for the
mechanism.

## Two pin stores — clarifying note

There are **two distinct TOFU stores** in the codebase. Mixing them
caused friction in early reading; documenting here so the next
auditor is not surprised:

- **`Trust_pin`** (`peer_review.ml`) — peer-pass artifact identity
  pinning. Stores `{pubkey, first_seen, last_seen}` per alias.
  Atomic `pin_rotate` with operator-attestation gate. Audit-logs on
  rotate success + unauth attempt. Path: `.c2c/peer-passes/pinned-keys.json`.
- **`Broker.known_keys_ed25519` / `_x25519`** (`c2c_mcp.ml`) —
  relay-e2e envelope TOFU. Flat `(alias, pk_b64)` Hashtbl + min-
  version sidecar. NO `first_seen` / `last_seen`. NO operator-
  attested rotate. Audit-logs only on mismatch. Path: `relay_pins.json`.

Design §5 ambiguously says "pin_rotate already exists from #432
TOFU 4/5", but #432 TOFU 4/5 added operator-attestation to
`Trust_pin.pin_rotate`, not to the relay-pin store. So §5.2 is
either (a) factually wrong about which store has rotate, or (b)
calling for parity — bringing rotate semantics from Trust_pin over
to the relay-pin store.

For this slice's purposes the distinction is: **the security
mechanism CRIT-2 needs is fully present.** A non-relay-pin operator
surface is a usability follow-on, not a security gap.

## Relay-side vs broker-side TOFU — clarification

The dispatch prompt suggested a relay-side TOFU might be a separate
thing the relay process does on forwarded envelopes. **It is not.**

`relay.ml` and `relay_forwarder.ml` treat envelope `content` as
opaque:

- `relay_forwarder.build_body` (forwarder.ml:21) just forwards
  `content` verbatim in a JSON wrapper.
- `relay.ml` has zero references to `Relay_e2e.*`, `from_ed25519`,
  `envelope_version`, or `verify_envelope_sig`.

The crypto end-to-end boundary is sender-broker → wire (via relay,
which is just transport) → receiver-broker. The receiver-broker is
where TOFU verify + pin happens, in `c2c_mcp.ml decrypt_envelope`.
There is no separate "relay verifies envelope sigs of forwarded
messages" code path that needs its own pin store; the relay does
not look inside envelopes at all.

(A future design could add relay-side envelope verify as
defense-in-depth, but that's net-new architecture, not in this
design doc, and explicitly out of scope of CRIT-2 as written.)

## Recommended next moves

### This slice — REPORT RESOLVED, no impl

The CRIT-2 security mechanism is shipped. The remaining gaps are
non-security-class follow-ons. Recommend slate report
"CRIT-2 mechanism shipped via Slice A+B+B-min-version+E follow-up;
audit-log first-seen and operator surfaces are observability

### Suggested follow-on slices

Each independent, splittable, safe to schedule on their own
cadence. **None block CRIT-2 closure.**

1. **`relay_e2e_pin_first_seen` audit log line** (~25 LoC) — mirrors
   `log_relay_e2e_pin_mismatch` shape, fires inside the
   `pin_ed25519_sync |> ignore` branch in `decrypt_envelope`. Nice-
   to-have for forensic completeness; the mismatch event already
   gives operators what they need for active-attack response.

2. **Slice D operator surfaces** (~80-110 LoC):
   - `c2c relay-pins list` — read pin store, format alias / pinned
     key (truncated b64) / min-version. Read-only, safe.
   - `c2c relay-pins delete <alias>` — atomic pin remove + audit
     log `relay_e2e_pin_deleted`. Operator-attested if we want
     symmetry with `Peer_review.pin_rotate`; or unattested if we
     decide local operator already has filesystem-write access to
     `relay_pins.json` and gating delete is theatre.
   - `c2c relay-pins rotate <alias>` — only if operators ask for
     it. Filesystem-write is the simpler escape hatch; deleting +
     letting next message TOFU-pin is functionally equivalent.

3. **§7.1 strict v2-without-field rejection at parse** (~10 LoC) —
   add a parse-time check in `decrypt_envelope` that an
   `envelope_version >= 2` envelope without `from_ed25519` is
   rejected as `Failed`. This is currently defended in practice via
   min-version pin (after first v2 contact, v1 envelopes from same
   alias get rejected as version-downgrade), but the literal
   pre-pin "v2 with no field is malformed" check is missing.
   Worth adding for defense-in-depth; not a CRIT-2 blocker.

4. **Cross-host divergence test** (~50 LoC, optional) — stand up
   two broker roots in one test, deliver the same wire through
   both, assert independent pin state.

5. **Slice C strict-v2 verifier flip** — separate operations
   slice, design says deferred to ops sign-off after soak.
