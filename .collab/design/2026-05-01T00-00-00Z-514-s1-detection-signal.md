# #514: MCP Self-Restart on Stale Broker Root — Slice 1 (Detection Signal)

## Problem Statement

The c2c MCP server resolves its broker root via `C2c_repo_fp.resolve_broker_root()`
at startup. The parent `c2c start` process also resolves it before spawning the MCP
server child, and passes the result via the `C2C_MCP_BROKER_ROOT` environment variable.

When `C2C_MCP_BROKER_ROOT` is **not set** (i.e. the parent resolved the default path
based on the git fingerprint), the MCP server uses that env value directly. If the
git remote URL changes between when the parent started and when the MCP server was
spawned, or if the MCP server process inherits a stale `C2C_MCP_BROKER_ROOT` from
a parent whose git context differs from its own, the MCP server writes to a
different broker directory than what it would compute fresh — splitting registration
and inbox from peers.

Agents can also be started with `c2c start` from a git clone whose `remote.origin.url`
has changed since the last run, causing the same split-brain.

## Detection Signal (this slice)

The detection runs **once** at MCP server startup, before any broker I/O.

### Algorithm

In `run_inner_server ~broker_root` in `c2c_mcp_server_inner.ml`:

1. Compute `canonical = C2c_repo_fp.resolve_broker_root ()`.
   This mirrors the parent's resolution exactly — same git fingerprint, same env
   fallbacks.
2. Compute `from_env = Sys.getenv_opt "C2C_MCP_BROKER_ROOT"`.
3. **Only check when `from_env` is `None`** — if the caller explicitly set
   `C2C_MCP_BROKER_ROOT`, we trust the override (it's intentional, not stale).
4. If `from_env = None` and `broker_root <> canonical`:
   - Log `[c2c] WARNING: stale broker_root detected` with details.
   - Exit the MCP server with exit code **42** (chosen as non-conflicting,
     not otherwise used by the server). The parent `c2c start` catches this
     and triggers a restart with the freshly-resolved broker root.

### Exit Code 42 Convention

`run_inner_server` already has a `try/with` wrapper that prints a "FATAL" backtrace.
We add a new `Stale_broker_root` variant to `C2c_mcp_server_inner` and catch it
above that wrapper so the exit is clean (not a crash backtrace).

The outer `c2c start` process (in `run_outer_loop`) checks the child exit status.
On 42, it re-invokes `start_mcp_inner` with the **currently computed** broker root
(fresh resolution) instead of the inherited one.

### Edge Cases

- **`C2C_MCP_BROKER_ROOT` is set to a relative path**: parent resolves it to
  absolute before passing. If the MCP server's CWD differs from the parent's CWD,
  the path could be wrong — but this is an operator misconfiguration, not staleness.
  We do NOT treat this as stale (the explicit env override wins).
- **Nested `c2c start` (c2c starting another c2c)**: the inner `c2c start` would
  have its own `C2C_MCP_BROKER_ROOT` set by its parent. The inner MCP server would
  see `from_env = Some(...)` and skip the check. Correct.
- **MCP server is the standalone `c2c-mcp-server` binary**: it resolves
  `broker_root` itself via `C2c_repo_fp.resolve_broker_root()` and does NOT receive
  it from a parent. The detection path is identical (from_env = None, compares
  against freshly-computed canonical) and correctly exits 42 on mismatch.

### Files Changed

```
ocaml/server/c2c_mcp_server_inner.ml  — stale detection + exit 42
ocaml/c2c_start.ml                     — catch exit 42 in run_outer_loop, re-invoke
```

## Slice 2 (separate): Restart Path

S2 implements the restart trigger in the outer `c2c start` process:
- In `run_outer_loop`, after the child MCP server exits with code 42,
  re-compute `broker_root` and re-launch the MCP server with the fresh value.
- S2 also handles the case where the MCP server is the standalone binary
  (`c2c-mcp-server` run directly) — the detection is the same, but the
  restart path differs (the binary just exits 42; a wrapper or init system
  would handle restart).
