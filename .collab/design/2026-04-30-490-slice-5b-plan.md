## Why this is its own slice

5a shipped the **verdict**-file path (the half that unblocks #142 e2e:
reviewer writes verdict, hook reads it, race avoided). This slice fills
in the **pending**-file half: hook records what's pending so reviewers
can `approval-list` independent of the awareness DM, plus inspect
details with `approval-show <token>`.

Splitting let 5a be reviewed cleanly without the operator-UX surface
bloating the diff.

## Scope

1. **CLI: `c2c approval-pending-write`** — bash-callable, writes
   `<broker_root>/approval-pending/<token>.json` from flags. Avoids
   bash needing to know broker-root resolution rules. Flags:
   `--token T` `--tool-name N` `--tool-input J` `--reviewer R`
   `--timeout S` `--agent-alias A` (defaults to current session
   alias, falls back to `unknown`).

2. **Hook script update** — in `c2c_kimi_hook.ml`'s embedded
   `approval_hook_script_content`, call
   `c2c approval-pending-write ...` BEFORE the DM send. Failure
   non-fatal (legacy DM path still works).

3. **CLI: `c2c approval-list`** — scans `<broker_root>/approval-pending/`
   and `<broker_root>/approval-verdict/`, prints a table of pending
   approvals. `--json` for machine readable. Exit 0 always (empty list
   is fine).

4. **CLI: `c2c approval-show <token>`** — pretty-prints the pending
   JSON for a single token. Exit 1 if not found.

5. **Tests** — alcotest cases for the new module functions. Smoke test
   via built binary: hook calls `approval-pending-write`, file lands,
   `approval-list` shows it, `approval-show` prints it, then
   `approval-reply` consumes (5a path) → `approval-list` shows
   verdict-recorded, then await-reply cleanup deletes both files.

## Out of scope (deferred to 5c)

- Notifier-daemon filter for `ka_*` DMs (defense-in-depth — for now
  the hook DM still goes through normal channels, which is fine
  because slice 5a means the verdict no longer relies on reading those
  DMs back).
- TTL cleanup for stale pending/verdict files. Hook + await-reply
  best-effort cleanup is sufficient v1.

## Drive-by from 5a smoke

Document in operator runbook that `c2c approval-reply` requires `--`
to pass reason text containing leading hyphens through Cmdliner's
positional parser:

    c2c approval-reply --reviewer ALIAS -- TOKEN deny because ...

Either fix Cmdliner option-parsing (consume rest as reason after
verdict positional) or document the workaround. Probably the simpler
fix is to make `--reason "..."` an explicit flag and treat positional
words as legacy-only.
