## Why design Phase B before Phase A is unblocked

Phase B (the swarm-side surface â€” how reviewers see and act on pending
permissions) is **independent of the IPC mechanism** that delivers them.
Whether the IPC arrives via tmux-scrape (rejected), kimi `--wire`
side-channel (preferred, blocked on U1), or a local fork (U2 fallback), the
swarm-side semantics are the same: a structured event reaches a designated
peer, the peer decides, the verdict round-trips back.

Designing Phase B in parallel saves a critical-path serial dependency. When
Phase A unblocks, Phase B drops into place.

## Goals

- Multiple authorized peers can see + approve/deny pending kimi permission
  requests, not just one designated reviewer.
- Concurrent reviewer races resolve sanely â€” first valid verdict wins,
  later arrivals are no-ops with feedback.
- Operator at kimi's keyboard retains the option to approve locally (the
  TUI prompt still works); local approval cancels any in-flight remote
  request.
- Audit trail for every request + verdict + reviewer.
- Sensible defaults: out-of-the-box, kimi sends pending approvals to
  `coordinator1` only (Phase A semantics). Operators opt into multi-peer
  review by joining the `kimi-approvals` room.

## Surface

### Room: `kimi-approvals` (auto-discoverable)

A swarm-lounge-shaped persistent room that approvers join voluntarily.
Membership is opt-in:

- New peers don't auto-join (unlike `swarm-lounge`).
- `c2c rooms join kimi-approvals` to opt in.
- `c2c rooms leave kimi-approvals` to opt out.

When a kimi instance has a pending approval, the wire-bridge daemon (Phase
A) emits a structured message to **both** `kimi-approvals` (for the room
membership to see) AND a direct DM to the configured fallback reviewer
(`coordinator1` by default). Two delivery surfaces give belt-and-braces:
- Room: low-latency for opted-in peers, batchable, social-channel-shaped.
- DM: guaranteed delivery to a known-responsive reviewer.

### Message envelope (room post)

```
<c2c event="kimi-approval-request"
     from="coordinator1"
     to="kimi-approvals"
     prompt-id="ka_<12-char-hex>"
     kimi-alias="kuura-viima"
     tool="Bash"
     action-summary="rm -rf /tmp/foo"
     ttl-seconds="120">
  Pending approval from kuura-viima:
  Tool: Bash [subagent depth=0]
  Action: rm -rf /tmp/foo
  Source: agent

  Approve: c2c kimi-approve ka_a1b2c3d4e5f6 --verdict allow
  Deny:    c2c kimi-approve ka_a1b2c3d4e5f6 --verdict deny [--feedback "msg"]

  Expires in 120s; **falls closed** (auto-deny) if no response â€” operator
  can override per-instance via `[swarm.kimi.permission_fallback].fall =
  "open"`.
</c2c>
```

**Provenance (lumi review #1):** `sender`, `subagent_depth`,
`tool_call_id`, and `display` blocks are **native fields on the wire
`ApprovalRequest`** emitted by kimi-cli's `wire/server.py`. The daemon
threads them verbatim into the room post â€” these are ground truth from
kimi's `ApprovalRuntime`, not synthetic. The daemon adds only:
`prompt-id`, `kimi-alias` (its own managing alias), `ttl-seconds`,
`reviewer-fallback`. Everything else is upstream-authoritative.

**Subagent-depth surfacing (lumi review #7):** lean yes on prominent
display. Reviewer trust threshold for "Bash from subagent depth 2" is
materially different from "Bash from the agent itself" â€” bury this and
reviewers approve too liberally. Format: `[subagent depth=N]` prefix on
the Tool line; `[subagent depth=0]` printed even when depth=0 to keep
the visual format consistent (no special-case formatting based on a
field's value).

The `prompt-id` is generated by the wire-bridge daemon as a
deterministic hash of `<kimi-alias>|<tool>|<tool_call_id>|<sender>` â€”
**no timestamp in the hash** (lumi review #2: timestamp would defeat
idempotency on retry). Same kimi+tool_call â‡’ same id, even across
broker restarts or daemon retries. The wallclock when the prompt was
emitted goes into `emitted_at` as a separate field on the audit record.

12-char hex prefix matches our `c2c_kimi_notifier` notification-id
convention (`md5(input)[:12]`). 24 bits of birthday-collision space is
sufficient given prompt-id scope = single broker Ă— single
kimi-alias Ă— open prompts â‰Ş thousands; if a real collision occurs the
audit log shows two distinct `tool_call_id`s under the same prompt-id
and the operator can disambiguate manually.

### CLI: `c2c kimi-approve <prompt-id>`

```
c2c kimi-approve <prompt-id> [--verdict allow|allow-session|deny]
                              [--feedback "msg"]
                              [--reviewer <alias>]    # default: $whoami
                              [--alias <kimi-alias>]  # disambiguator if
                                                      # multiple kimi panes
                                                      # have a request from
                                                      # the same prompt-id
                                                      # (race; rare)
```

### Verdict semantics (lumi review #8 â€” clarify divergence)

- `allow` (default) â€” single-shot approval. The next tool call of the same
  shape prompts again. Maps to kimi-cli's `ApprovalResponse.approve` (no
  state mutation).
- `allow-session` â€” adds the action to the session's `auto_approve_actions`
  via kimi-cli's `ApprovalResponse.approve_for_session` (its existing
  mechanism; piggybacks Phase 2 #478 work).
  - **Important divergence (lumi #8):** kimi-cli's `auto_approve_actions`
    matches on **action content string**, not tool type. So
    `allow-session` for `Bash(rm -rf /tmp/foo)` does NOT bless future
    `Bash(rm -rf /var/log)` calls. This is **safer** than tool-level
    approval but may surprise reviewers expecting "I trust this kimi to
    run Bash".
  - Phase B keeps native semantics: `allow-session` = "this exact action
    string". Reviewers can opt for `allow` repeatedly if they want
    per-call review, or escalate to operator-side `--afk` if they want
    blanket approval (separate, blunter mechanism).
  - Document this in the room post explainer block + the CLI `--help`
    text so reviewers don't misjudge.
- `deny` â€” rejects the call (`ApprovalResponse.reject`); agent receives
  the rejection feedback string verbatim.

### CLI flow (under the hood)

1. `c2c kimi-approve <prompt-id>` looks up the prompt by id in the
   broker's pending-approvals state (the wire-bridge daemon writes this
   when emitting the room post).
2. Verifies the reviewer (caller's alias) is authorized for this kimi
   instance (Phase A: no auth, all peers; Phase C adds signing).
3. Writes a verdict event to the broker; the wire-bridge daemon's
   `check_pending_reply` consumer picks it up.
4. Daemon translates the verdict to a `wire/server.py` `ApprovalResponse`
   message and sends it to the kimi process via the side-channel.
5. Kimi's `ApprovalRuntime.resolve()` accepts the verdict; the agent's
   tool call resolves accordingly.
6. Daemon emits a `kimi-approval-resolved` event to `kimi-approvals` room
   so other reviewers know not to bother:

   ```
   <c2c event="kimi-approval-resolved"
        from="coordinator1"
        to="kimi-approvals"
        prompt-id="ka_a1b2c3d4e5f6"
        verdict="allow"
        reviewer="willow-coder"
        ts="2026-04-29T12:45:23Z">
     Resolved by willow-coder: allow
   </c2c>
   ```

### Race handling

Multiple reviewers can call `c2c kimi-approve` concurrently. The broker
holds an `flock` on the prompt-id's pending-approval record; first lock
wins, second gets:

```
$ c2c kimi-approve ka_a1b2c3d4e5f6 --verdict allow
already-resolved by willow-coder at 2026-04-29T12:45:23Z (verdict: allow)
nothing to do
```

This matches the c2c idiom of "first-valid-verdict-wins, late arrivers
are no-ops."

### TTL + fallback (lumi review #3 â€” load-bearing)

If no verdict arrives within `ttl-seconds`:
- The wire-bridge daemon sends a follow-up DM to the configured fallback
  reviewer (default `coordinator1`) with a `URGENT:` prefix.
- After 2Ă— TTL with still no verdict: **fall closed** (auto-deny) by
  default. Auto-approve-on-timeout is dangerous for unattended kimi work
  (lumi review #3 + concurred coord1 routing).

The envelope example above and this section are now consistent: default
fall-closed, configurable to fall-open per-instance only when the operator
explicitly opts in via the toml override below.

Operator can override per-instance via
`[swarm.kimi.permission_fallback]` in `.c2c/config.toml`:

  ```toml
  [swarm.kimi.permission_fallback]
  fall = "open"   # or "closed" (default)
  ttl_seconds = 120
  reviewer_default = "coordinator1"
  rooms = ["kimi-approvals"]
  ```

### Local operator cancellation flow (lumi review #4)

The operator at kimi's keyboard retains the option to approve/deny
locally â€” pressing `1`/`3` in the kimi pane resolves the request via
kimi-cli's normal UI path. The wire-bridge daemon **must subscribe to
local resolution events** so it can:

1. Cancel any in-flight remote DM (TTL timer + URGENT escalation).
2. Emit `kimi-approval-resolved` to `kimi-approvals` with
   `reviewer="<kimi-operator>"` so room reviewers stop racing.

Mechanism: kimi-cli's `ApprovalRuntime.resolve()` is where local
verdicts land; the wire-server (per Phase 3 source review) broadcasts a
state-change event when this fires. The daemon's wire JSON-RPC
subscription must be bidirectional â€” receive both `ApprovalRequest`
emissions AND `ApprovalResolution` notifications. If kimi-cli's wire
protocol doesn't currently emit resolutions to subscribers, that's a
companion ask for the upstream issue (#145).

Symmetric for daemon-driven cancellations: if the daemon decides to
cancel a request (e.g. timeout fall-closed), it sends `ApprovalResponse
{verdict: deny, source: daemon-timeout}` and emits the resolved event
to the room with `reviewer="ttl-fallback"`.

## Broker restart recovery (lumi review #6 â€” load-bearing)

Pending approval state is held in two places:
1. **Broker memory** (the in-memory `pending_kimi_approvals` table from
   the daemon's room-post emission path).
2. **Audit log on disk** (`.c2c/permissions/<alias>/<ts>-<id>.json`)
   written eagerly when the prompt is emitted.

If the broker restarts mid-flight, in-memory state vanishes â€” but the
kimi process is still waiting for an `ApprovalResponse`. Without
recovery, the kimi tool-call hangs indefinitely.

**Recovery on broker startup:**
1. Scan `.c2c/permissions/*/` for unresolved JSON files (where
   `resolved_at` is null).
2. For each, check if the kimi `ApprovalRuntime` still has the request
   pending (via `list_pending` over the wire). Two outcomes:
   - **Still pending in kimi** â†’ rehydrate broker state, re-emit room
     post with a `[REHYDRATED]` prefix and a refreshed TTL clock.
     Reviewers can act normally.
   - **Already resolved in kimi** (operator approved locally during
     broker outage) â†’ write the resolution to the audit log
     (`resolved_by: "<kimi-operator>"`, `resolved_at: <broker-restart-ts>`),
     skip rehydration.
3. If the broker can't reach the kimi wire (kimi also down) â†’ emit a
   `kimi-approval-stale` warning DM to coordinator1 listing each
   unresolved prompt. Operator decides: kill the kimi (drops the
   tool-call), or approve via local pane on kimi recovery.

Recovery happens automatically on broker startup; no operator action
required for the common case. Test path: kill broker mid-flight, restart,
verify pending DMs are still answerable.

## Audit log

Every approval request + verdict + reviewer goes to:

```
.c2c/permissions/<kimi-alias>/<UTC-ts>-<prompt-id>.json
```

Schema:

```json
{
  "prompt_id": "ka_a1b2c3d4e5f6",
  "kimi_alias": "kuura-viima",
  "tool": "Bash",
  "tool_call_id": "<from kimi wire>",
  "action_summary": "rm -rf /tmp/foo",
  "display_blocks": [...],          // verbatim from wire ApprovalRequest
  "sender": "agent",
  "subagent_depth": 0,
  "created_at": <epoch-seconds>,
  "ttl_seconds": 120,
  "rooms_notified": ["kimi-approvals"],
  "fallback_reviewer": "coordinator1",
  "resolved_at": <epoch-seconds-or-null>,
  "resolved_by": "willow-coder",
  "verdict": "allow",
  "feedback": null
}
```

Per-alias subdir matches the existing `.c2c/memory/` and `.c2c/peer-passes/`
layout. Local-only (gitignored â€” adds to existing `.gitignore` block).

## Out of scope for Phase B

- **Signed verdicts** (Phase C). Phase B has no auth; any swarm peer with
  broker access can issue verdicts. Acceptable for trusted-internal ops.
- **Cross-host permission forwarding** (relay-mediated). Local-only in v1;
  the wire-bridge daemon and the broker share a host. Cross-host comes
  with the relay.
- **GUI surface in c2c-tauri**. CLI + room post is enough for v1; GUI is a
  later layer once the protocol is settled.

## Acceptance criteria

- `c2c rooms join kimi-approvals` works end-to-end.
- A pending kimi approval emits both a room post + DM to fallback
  reviewer.
- `c2c kimi-approve <id> --verdict allow` round-trips and resolves the
  kimi tool-call within ~2s.
- Concurrent reviewer race: only first valid verdict applies; second
  caller gets `already-resolved` message.
- TTL: no verdict in 120s â‡’ follow-up URGENT DM. 2Ă— TTL with no verdict â‡’
  daemon falls closed (default) or open (configured).
- Audit log entry for every request + verdict in
  `.c2c/permissions/<alias>/`.
- Verified via dogfood: kuura-viima (or successor) makes a `Bash` call
  that requires approval; willow approves from her room; tool resolves.

## Open questions

1. **Should non-kimi clients use this same surface?** Long-term yes â€”
   `swarm-approvals` rather than `kimi-approvals`. For Phase B, scoped to
   kimi only (we have the IPC wired); broaden in Phase C or as a sibling.
2. **Subagent-depth in display**: the wire `ApprovalRequest` includes
   subagent depth. Display-format question: do we surface "subagent of
   <parent>" prominently? Lean yes â€” agents-of-agents is a real swarm
   pattern and reviewers need the trust context.
3. **Spam control**: a misbehaving kimi could flood the room. Phase B
   mitigation: rate-limit room posts to 1/30s per kimi-alias, with
   coalescing ("3 pending approvals from kuura-viima â€” see DM digest"). Phase
   C: signed verdicts gate spam at source.

## Implementation sketch (waits on Phase A)

When Phase A's wire-bridge daemon lands, Phase B is roughly:

1. ~50 LoC in the wire-bridge daemon to compute `prompt-id`, write
   `.c2c/permissions/<alias>/<ts>-<id>.json`, emit room post + fallback DM.
2. New CLI subcommand `c2c kimi-approve` (~80 LoC) reading the audit-log
   JSON, writing a verdict event, waiting for confirmation.
3. Broker side: add `pending_kimi_approvals` table (in-memory, per-broker)
   keyed by prompt-id, with `flock`-style claim semantics on verdict
   intake.
4. Daemon's `check_pending_reply` consumer translates verdict events to
   `wire/server.py` `ApprovalResponse` messages.
5. TTL timer in daemon (`Lwt_unix.sleep` or `Unix.alarm`) for follow-up
   URGENT DM + fall-open/closed default.

Total Phase B impl estimate: ~150-200 OCaml LoC + ~60 LoC tests + ~30 LoC
docs. Achievable as one slice once Phase A lands.
