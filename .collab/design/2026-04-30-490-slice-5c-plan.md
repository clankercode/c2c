## Scope

1. **Notifier-daemon filter for `ka_*` DMs** — `ocaml/c2c_kimi_notifier.ml`
   pushes inbound DMs to the kimi pane via tmux send-keys. Approval
   verdicts use `c2c approval-reply` and the file-based side-channel,
   so any DM whose body matches the legacy verdict shape
   (`^\s*ka_[A-Za-z0-9_-]+\s+(allow|deny)\b`) MUST NOT be tmux-injected
   into kimi's input queue. Suppress notifier delivery; the verdict
   file is the canonical handoff.

   Defense-in-depth: even if the reviewer mistakenly uses legacy
   `c2c send <kimi> "<TOKEN> allow"`, the notifier doesn't poison
   kimi's user-input stream. The hook's `await-reply` may still pick
   it up via the legacy fallback (slice 5a kept that path) so the
   approval still resolves; we just don't queue it as user input.

2. **TTL cleanup — `c2c approval-gc` subcommand** —
   - Removes `approval-pending/<token>.json` if `timeout_at` (parsed
     from JSON) is older than `now()`.
   - Removes `approval-verdict/<token>.json` if `mtime` is older than
     a configurable max age (default 1h).
   - Dry-run by default (`--apply` to actually remove). `--json` for
     machine readable. Prints classification per file.
   - Optionally invoked implicitly at the end of `approval-list` runs
     gated by `--gc` flag (low-frequency).

3. **Tests** — alcotest cases:
   - notifier filter: feed verdict-shape body through the filter and
     verify the suppress-tmux path. Feed normal DM and verify
     pass-through.
   - approval-gc: write expired pending + recent verdict, verify gc
     removes only the right files, idempotent.

## Out of scope

- Cross-host TTL coordination (relay sync of pending state) — v2.
- Multi-reviewer / quorum — design doc deferred.
